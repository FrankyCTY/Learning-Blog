OOP vs FP

Inheritance vs Composition

OOP focuses on inheritance from a parent class to the child classes. (What it is)

FP focuses on composition that handle objects or handle a piece of data by composing them from different say pure functions. (What it has)

Most programmers tend to use composition over inhertiance, but why?

Downside of inheritance

Inheritance helps to maximize the reusability of the code but it fixes the code and can lead to a lot of problems in the long run by causing rippling effect to the child classes.

Because it causes tightly coupling.

A simple inheritance demostration
            

                User
                    Watcher
                    Character
                        Elf
                            Young Elf
                        Orge

What if:

- We only want the 'Young Elf' class to have one behaviour (sleep()), but the parent classes actually inherit a bunch of behaviours to it that it doesn't want.

- We want the 'Watcher' class to move up to be the parent class of the 'User' class.

Everything breaks

Composition can solve this

We make use of different combination of functions.

Passing an object and return a new object with required behviours.

Demonstration using composition
            
                function getAttack(character) {     // Pure function
                    return Object.assign({}. character, {attackBehaviour: () =>{}}) // Add attack behaviour and return the new object
                }

                function Elf(name, weapon, type) {
                    lef elf {
                        name,
                        weapon,
                        type
                    }

                    return getAttack(); // Return a Elf instance that has attack behaviour
                }

Now we can compose the object whatever we want.

The inheritance is gone, and we compose the 'Elf' object by composing the pure function behaviour and return a new Elf object.

NOTE: We didn't use constructor function directly, because we want to build this Elf function imperatively so that we can manipulate every behaviour we need to add before returning the Elf object.

OOP vs FC

Object Oriented Programming:

Encapsulate data and behaviours at the same location

Object - First Class Citizen

Many operations on fixed data

Stateful

Side Effect

Imperative

Functional Programming:

Data and behaviours are different and should be kept separately.

Function - First Class Citizen

Few operations on common data

Stateful

Pure - NO side effect

Declarative

When to use them?

Object Oriented Programming:

When we have many things (characters) with few operations.

Functional Programming:

When we have a few things that have a lot of different operations on them.

Parallel programming
